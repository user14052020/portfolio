# Что Такое Язык Программирования

Это следующая фундаментальная глава курса.

После неё слова `язык программирования`, `синтаксис`, `runtime`, `стандартная библиотека`, `JavaScript`, `TypeScript` и `Python` уже не должны смешиваться в одну кучу.

Именно на этой базе дальше будут нормально читаться:
- frontend на React и Next.js;
- backend на Python;
- конфиги вроде `next.config.mjs`;
- файлы `*.ts`, `*.tsx`, `*.py`;
- зависимости, пакеты и фреймворки.

## Что Это Академически

### Язык Программирования

Язык программирования — это формальная знаковая система, предназначенная для записи алгоритмов, структур данных и правил управления вычислительным процессом таким образом, чтобы эти инструкции могли быть обработаны вычислительной системой.

Если говорить короче и строже:
- язык задаёт правила записи кода;
- язык задаёт допустимые конструкции;
- язык задаёт способы выражать данные, вычисления, ветвления, функции, модули и типы.

### Синтаксис

Синтаксис — это набор формальных правил, определяющих, как именно можно записывать конструкции языка.

Например:
- где ставятся скобки;
- как объявляются функции;
- как оформляются импорты;
- как пишутся условия;
- где допустимы двоеточия, точки с запятой, отступы и ключевые слова.

### Семантика

Семантика — это значение записанной конструкции, то есть что именно программа будет делать, когда эта запись будет выполнена.

Пример:
- синтаксис отвечает на вопрос “можно ли так написать?”;
- семантика отвечает на вопрос “что это означает?”.

### Runtime

Runtime — это среда выполнения, в которой код реально запускается и выполняется.

Runtime отвечает за то, как язык живёт в реальности:
- как запускается код;
- как устроены модули;
- как выполняются функции;
- как организованы ошибки;
- как программа взаимодействует с файлами, сетью, памятью и системными API.

### Стандартная Библиотека

Стандартная библиотека — это набор модулей, функций, типов и инструментов, которые поставляются вместе с языком или его основной реализацией и доступны без установки внешних пакетов.

Например, стандартная библиотека обычно даёт:
- работу со строками;
- даты и время;
- файловую систему;
- сериализацию;
- сетевые функции;
- математические операции;
- обработку ошибок;
- структуры данных.

## Что Это Простыми Словами

Если по-человечески:

- язык программирования — это способ объяснить компьютеру, что делать;
- синтаксис — это правила грамматики этого способа;
- runtime — это “сцена”, на которой этот язык реально оживает;
- стандартная библиотека — это набор встроенных инструментов, который уже лежит у вас в ящике, ещё до установки сторонних пакетов.

Очень важно не путать:

- язык;
- runtime;
- библиотеку;
- фреймворк;
- пакетный менеджер.

Это разные уровни.

Например:
- `TypeScript` — это язык;
- `Node.js` — это runtime для JavaScript вне браузера;
- `FastAPI` — это фреймворк;
- `Next.js` — это фреймворк;
- `npm` и `pip` — это пакетные менеджеры.

## Почему Это Реализовано Именно Так

В нашем проекте используются сразу несколько языковых слоёв, потому что у системы разные задачи.

### Почему На Фронте JavaScript / TypeScript

Потому что браузерная экосистема строится вокруг JavaScript.

Именно поэтому:
- браузер понимает JavaScript;
- React и Next.js работают в JavaScript-экосистеме;
- TypeScript нужен как надстройка, которая помогает писать более надёжный frontend-код.

### Почему На Бэке Python

Потому что Python хорошо подходит для:
- HTTP API;
- интеграций;
- работы с данными;
- AI-инфраструктуры;
- быстрой серверной разработки;
- богатой экосистемы вокруг LLM и image generation.

### Почему Языков Несколько

Потому что язык выбирают не “один раз навсегда для всего мира”, а под конкретную роль в системе.

В этом проекте это выглядит так:
- frontend: TypeScript и JavaScript-экосистема;
- backend: Python;
- конфигурация frontend runtime: JavaScript module в `next.config.mjs`;
- инфраструктура: YAML, `.env`, markdown и shell-команды как вспомогательные языки описания среды.

## Какие Anti-Patterns Рядом Существуют

### Anti-pattern 1. Думать, что TypeScript — это отдельный runtime

Это неверно.

TypeScript не исполняется в браузере напрямую как “особый отдельный мир”.

Практически:
- разработчик пишет `*.ts` и `*.tsx`;
- tooling преобразует их в JavaScript;
- дальше уже работает JavaScript runtime.

То есть TypeScript — это язык разработки с типами и tooling-поддержкой, а не отдельная браузерная вселенная.

### Anti-pattern 2. Думать, что Python = Django

Нет.

Python — это язык.
Django — это фреймворк на Python.

В этом проекте backend написан на Python, но не на Django. Здесь используется FastAPI.

### Anti-pattern 3. Путать язык и фреймворк

Это разные уровни:
- `Python` — язык;
- `FastAPI` — фреймворк;
- `JavaScript` — язык;
- `React` — библиотека;
- `Next.js` — framework/runtime layer поверх React.

### Anti-pattern 4. Думать, что расширение файла полностью объясняет смысл

Например:
- `.ts` и `.tsx` говорят, что это TypeScript-код;
- `.mjs` говорит, что это JavaScript module;
- `.py` говорит, что это Python.

Но реальная роль файла определяется не только расширением, а ещё и местом в архитектуре.

Например:
- `apps/frontend/src/shared/api/client.ts` — это клиент API;
- `apps/frontend/src/app/page.tsx` — это страница;
- `apps/backend/app/main.py` — entrypoint backend;
- `apps/backend/app/core/config.py` — конфигурация backend.

### Anti-pattern 5. Считать, что “язык = всё, что нужно знать”

На практике одного знания языка мало.

Нужно отдельно понимать:
- runtime;
- стандартную библиотеку;
- зависимости;
- пакетный менеджер;
- фреймворк;
- архитектуру проекта;
- среду выполнения.

## JavaScript, TypeScript И Python На Простом Уровне

### JavaScript

JavaScript — это высокоуровневый язык программирования общего назначения, исторически возникший как язык для браузера, но сегодня используемый и на сервере, и в tooling, и в сборке приложений.

Простыми словами:
- это основной язык браузерного мира;
- на нём строятся React, Next.js и огромная часть frontend-разработки;
- он может работать не только в браузере, но и вне браузера, например через Node.js.

### TypeScript

TypeScript — это язык, расширяющий JavaScript системой статической типизации и дополнительными средствами анализа кода, после чего такой код преобразуется в обычный JavaScript.

Простыми словами:
- это JavaScript “с дополнительной страховкой”;
- он помогает ловить ошибки раньше;
- он улучшает автодополнение, рефакторинг и читаемость больших проектов.

### Python

Python — это высокоуровневый язык программирования общего назначения, ориентированный на читаемость кода, выразительность и богатую экосистему библиотек.

Простыми словами:
- это язык, на котором удобно писать backend, скрипты, AI-интеграции и системную glue-логику;
- он популярен потому, что код на нём обычно быстро писать и поддерживать.

## Что Реализовано В Этом Проекте

### 1. Frontend Исходники Пишутся На TypeScript

Это видно по файлам:
- `apps/frontend/src/shared/api/client.ts`
- `apps/frontend/src/app/page.tsx`
- `apps/frontend/src/app/HomePageSurface.tsx`

Например:
- `apps/frontend/src/shared/api/client.ts` содержит функции API-клиента;
- `apps/frontend/src/app/page.tsx` — точка входа страницы;
- `*.tsx` означает TypeScript + JSX-разметку компонентов.

То есть frontend проекта пишется не на “чистом браузерном JavaScript без типизации”, а на TypeScript.

### 2. Frontend Runtime И Tooling Опираются На JavaScript-Экосистему

Хотя бизнес-код фронта в проекте пишется в `*.ts` и `*.tsx`, у нас есть и явный JavaScript-файл конфигурации:

- `apps/frontend/next.config.mjs`

Это хороший практический пример того, что JavaScript остаётся базой всей этой экосистемы:
- конфиги;
- build-tools;
- часть runtime-настроек Next.js;
- npm-скрипты и Node-окружение.

### 3. Backend Написан На Python

Это видно по файлам:
- `apps/backend/app/main.py`
- `apps/backend/app/core/config.py`
- `apps/backend/app/db/session.py`
- `apps/backend/app/services/*`

Например:
- `apps/backend/app/main.py` создаёт приложение FastAPI;
- `apps/backend/app/core/config.py` описывает настройки backend;
- `apps/backend/app/db/session.py` создаёт async engine и DB session.

То есть backend — это уже не JavaScript-мир, а Python-мир.

### 4. В Проекте Хорошо Видна Разница Между Языком И Framework

На фронте:
- язык: TypeScript / JavaScript;
- библиотека/фреймворк: React / Next.js.

На бэке:
- язык: Python;
- фреймворк: FastAPI.

Это очень важная связка. Иначе начинающий разработчик быстро начинает говорить вещами вроде:
- “я пишу на Next.js”;
- “у нас язык FastAPI”;
- “backend на Django”, когда речь вообще о Python.

Технически корректнее разделять:
- язык;
- framework;
- runtime;
- инфраструктуру.

### 5. В Проекте Видна Разница Между Языком И Runtime

Frontend:
- код пишется в TypeScript;
- затем живёт в JavaScript runtime браузера или Node/Next tooling.

Backend:
- код пишется на Python;
- затем запускается Python runtime внутри backend-контейнера.

Это особенно важно помнить, когда вы видите такие вещи:
- `docker-compose.yml` запускает `uvicorn` для Python backend;
- `package.json` управляет frontend-скриптами и Node tooling;
- `next.config.mjs` является JavaScript module;
- `client.ts` и `page.tsx` — TypeScript source files.

## Синтаксис, Runtime И Стандартная Библиотека На Простом Примере

Возьмём одну и ту же задачу: сложить два числа и вернуть результат.

### JavaScript

```js
function add(a, b) {
  return a + b;
}
```

### TypeScript

```ts
function add(a: number, b: number): number {
  return a + b;
}
```

### Python

```python
def add(a: int, b: int) -> int:
    return a + b
```

Что тут важно:
- задача одна и та же;
- смысл один и тот же;
- синтаксис разный;
- подход к типам разный;
- runtime будет разный.

## Разбор На Примере

### Простой Вымышленный Пример

Представим маленький проект `greeting-app`.

На JavaScript это мог бы быть файл:

```js
const name = "Mira";
console.log(`Hello, ${name}!`);
```

На TypeScript:

```ts
const name: string = "Mira";
console.log(`Hello, ${name}!`);
```

На Python:

```python
name = "Mira"
print(f"Hello, {name}!")
```

Во всех трёх случаях идея одна:
- есть данные;
- есть инструкция вывести приветствие;
- есть язык, который позволяет это записать.

Но среда выполнения и экосистема будут различаться.

### Пример Из Текущего Проекта

Если взять текущий проект, то картина выглядит так:

- `apps/frontend/src/shared/api/client.ts` — TypeScript-код, который строит HTTP-запросы;
- `apps/frontend/next.config.mjs` — JavaScript-конфиг frontend runtime;
- `apps/backend/app/main.py` — Python-код, который поднимает сервер;
- `apps/backend/app/core/config.py` — Python-код, который читает переменные окружения.

Получается важная инженерная мысль:

один проект может использовать несколько языков одновременно, и это нормально, если у каждого из них понятная роль.

## Что Ещё Можно Улучшить

Даже на уровне языков здесь уже можно заранее заложить хорошее инженерное мышление.

Например, дальше в курсе важно отдельно показать:
- как TypeScript помогает фронту на уровне доменных типов;
- как Python-код на бэке выигрывает от более чётких сервисных границ и исключений;
- где типизация на фронте сейчас помогает проекту, а где ещё может стать строже;
- как choice языка влияет на производительность, tooling и сопровождение;
- почему “удобный язык” не заменяет архитектуру.

## Команды

### Найти TypeScript и JavaScript файлы на фронте

```bash
rg --files apps/frontend | rg "\\.(ts|tsx|mjs)$"
```

### Найти Python файлы на бэке

```bash
rg --files apps/backend | rg "\\.py$"
```

### Посмотреть frontend package и понять, что это Node/JS-экосистема

```bash
Get-Content apps/frontend/package.json
```

### Посмотреть backend requirements и понять, что это Python-экосистема

```bash
Get-Content apps/backend/requirements.txt
```

### Проверить версии runtimes локально

```bash
node --version
python --version
```
