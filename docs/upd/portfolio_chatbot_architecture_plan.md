# План развития fashion stylist chat: архитектура, стабильность, масштабируемость

## Цель

Сделать проект устойчивым и предсказуемым с точки зрения архитектуры, чтобы бот:

- корректно отвечал на обычные текстовые сообщения как стилист-консультант;
- стабильно отрабатывал три предопределённые команды:
  - **Подобрать к вещи**;
  - **Попробовать другой стиль**;
  - **Что надеть на событие**;
- умел поддерживать диалог как **историк моды**, **практик-портной**, **стилист по силуэту и цвету**;
- использовал базу стилей как источник знаний;
- строил **разнообразные** и **эстетичные** промпты для генерации flat lay изображений;
- был удобен для дальнейшего развития: новые режимы, новые базы знаний, новые модели, новые воркеры.

---

## Текущие симптомы

### Проблема 1. Некорректные ответы на текстовые сообщения
Симптом: бот часто отвечает не в тему, слишком общо или логически нестабильно.

### Проблема 2. Команда «Подобрать к вещи»
Симптом: пользователь пишет описание вещи, бот что-то отвечает текстом, но генерация образа не происходит.

### Проблема 3. Команда «Попробовать другой стиль»
Симптом: следующая картинка почти повторяет предыдущую: те же цвета, похожие вещи, та же композиция.

### Проблема 4. Команда «Что надеть на событие»
Симптом: бот задаёт уточнение, пользователь конкретизирует, после чего наступает тишина вместо генерации.

---

## Главная архитектурная причина проблем

Сейчас поведение бота похоже на смесь нескольких режимов в одном общем чате без жёсткой state machine и без стабильного контракта между:

- **frontend intent / быстрыми командами**;
- **backend routing**;
- **session state**;
- **decision layer LLM**;
- **prompt builder**;
- **generation job orchestration**;
- **ComfyUI workflow**.

Из-за этого происходят типовые сбои:

1. intent теряется или угадывается заново;
2. follow-up сообщения обрабатываются как новый общий запрос, а не как продолжение сценария;
3. LLM принимает решение, но итоговый generation prompt теряет часть контекста;
4. генерация зависит от слишком общего prompt builder и даёт однообразные результаты;
5. система не имеет достаточной наблюдаемости: не видно, где именно сломался путь — в классификации, генерации решения, prompt builder, очереди или ComfyUI.

---

## Принципиальное архитектурное решение

Нужно перестать мыслить ботом как одним общим диалогом и разделить его на **4 режима**:

1. `general_advice`
2. `garment_matching`
3. `style_exploration`
4. `occasion_outfit`

У каждого режима должны быть:

- свой контракт входа;
- свои обязательные поля state;
- свои правила уточнений;
- свой переход в генерацию;
- свой prompt-building pipeline;
- свои критерии завершения.

Это не быстрый фикс, а базовая архитектурная рамка, которая позволит проекту расти без постоянного переписывания.

разбиваем на маленькие шаги и один за другим закрываем без пропусков, если задача большая делаешь остановку перед следующеей партией шагов, а не костыль, реализация должна быть с учетом масштабируемости и удобства поддержания

# Этап 0. Создать пайплайн пополнения базы стилей

Нужно делать не вручную по одному объекту, а через отдельный ingestion pipeline.
[Детально в style_ingestion_plan_detailed.md](./0-parser/style_ingestion_plan_detailed.md) 
[Как пользоваться в style_ingestion_operations.md](./0-parser/style_ingestion_operations.md) 

## 0.1. Компоненты парсера
Создать отдельный модуль, не связанный с chat runtime:
Подробнее здесь C:\dev\portfolio\docs\upd\style_ingestion_plan.md
- `style_source_registry`
- `style_scraper`
- `style_normalizer`
- `style_enricher`
- `style_validator`
- `style_db_writer`

## 0.2. Что делает пайплайн
1. Берёт запись из базы стилей;
2. ищет соответствующее описание на доверенных сайтах; 
3. парсит текст;
4. нормализует структуру;
5. извлекает признаки стиля;
6. сохраняет в БД;
7. логирует источник и дату обновления.

## 0.3. Архитектурное требование
Парсер не должен жить внутри chat backend. Это отдельный ingestion-процесс / cron / worker.

# Этап 1. Зафиксировать доменную модель и сценарии

## 1.1. Ввести единый контракт режима чата
Создать единый внутренний объект `ChatModeContext`, который должен жить в backend session state.

Пример полей:

- `active_mode`
- `requested_intent`
- `pending_clarification`
- `clarification_kind`
- `should_auto_generate`
- `anchor_garment`
- `occasion_context`
- `style_history`
- `last_generation_prompt`
- `last_generated_outfit_summary`
- `conversation_memory`

### Зачем это нужно
Сейчас часть информации хранится разрозненно или восстанавливается заново из текста. В масштабе это всегда приводит к логическим провалам.

---

## 1.2. Описать конечные state machine для 4 режимов

### `general_advice`
Путь:
- user message
- interpret
- answer text
- optionally suggest next action

### `garment_matching`
Путь:
- нажата команда
- **отключается таймаут ожидания 60 сек**
- бот просит уточнение о вещи
- пользователь отвечает
- система объединяет:
  - название команды
  - последний пользовательский ответ
  - дополнительный контекст профиля
- decision layer строит outfit brief
- создаётся generation job
- бот возвращает текст + изображение

### `style_exploration`
Путь:
- нажата команда
- из базы выбирается стиль или стиль приходит из контекста
- система достаёт последние style directions
- decision layer строит **новый** style brief с анти-повтором
- generation job
- текст + изображение

### `occasion_outfit`
Путь:
- нажата команда
- **таймаут 60 сек не применяется**
- бот задаёт уточнение
- пользователь отвечает
- ответ складывается в `occasion_context`
- если контекста хватает — сразу generation job
- если не хватает — ещё одно строгое уточнение
- затем генерация

### Ключевое требование
Ни один follow-up в этих трёх режимах не должен интерпретироваться как новый общий чат-запрос.

---

# Этап 2. Нормализовать frontend как тонкий UI, а не место логики

## 2.1. Quick actions должны передавать только режим и сообщение
Frontend не должен сам решать, запускать генерацию или нет по куче условий. Он должен передавать:

- `requested_intent`
- `message`
- `session_id`
- `asset_id` если есть
- `command_context` если это команда

### Почему
Если бизнес-логика размазана между фронтом и бэком, проект становится хрупким и плохо тестируемым.

---

## 2.2. Ввести отдельный тип payload для chat command
Например:

```ts
{
  session_id,
  requested_intent,
  command_name,
  message,
  asset_id,
  command_step,
}
```

### Преимущество
Так проще отлаживать сценарии и логировать поведение.

---

## 2.3. Убрать фронтовую зависимость логики от наличия asset
Для `garment_matching` пользователь может описывать вещь текстом. Наличие изображения должно быть опцией, а не обязательным условием.

---

# Этап 3. Переписать decision layer как явный orchestrator

## 3.1. Выделить orchestration service
Нужен отдельный уровень между API route и LLM, например:

- `stylist_chat_orchestrator.py`

Он должен решать:

- какой режим активен;
- хватает ли контекста для генерации;
- нужно ли уточнение;
- какую память и knowledge base подмешивать;
- какой prompt builder вызывать;
- создавать ли generation job;
- что возвращать пользователю сразу.

### Почему
Сейчас слишком много ответственности может лежать в одном сервисе. Это затрудняет развитие и тестирование.

---

## 3.2. Ввести явный `DecisionResult`
Каждый шаг оркестратора должен возвращать строго типизированный результат, например:

- `text_only`
- `text_and_generate`
- `clarification_required`
- `error_recoverable`

### Почему
Это устраняет «тишину» после уточнений.

---

# Этап 4. Исправить «Подобрать к вещи» как полноценный сценарий

## 4.1. Формализовать вход
После нажатия кнопки:

- бот отправляет специальное служебное сообщение: «Опиши вещь, к которой нужно подобрать образ»;
- это сообщение переводит session state в `garment_matching.awaiting_anchor_garment`;
- таймаут 60 сек не должен завершать сценарий.

## 4.2. После ответа пользователя формировать `anchor_garment`
Нужно выделять и сохранять:

- тип вещи;
- цвет;
- материал;
- фасон;
- сезонность;
- гендерный контекст, если нужен;
- формальность;
- силуэт.

Если данные неполные — задаётся **одно короткое уточнение**, а не общий стилистический ответ.

## 4.3. Генерация должна быть обязательным завершением режима
Если режим `garment_matching` и есть достаточный `anchor_garment`, система **всегда** должна переходить в generation job.

Не должно быть ветки «дать совет текстом и не генерировать» по умолчанию.

---

# Этап 5. Исправить «Что надеть на событие» как slot-filling flow

## 5.1. Ввести slot model для мероприятия
Минимальные слоты:

- `event_type`
- `location`
- `time_of_day`
- `season`
- `dress_code`
- `weather_context` (опционально)
- `desired_impression`

## 5.2. Уточнения должны идти по слотам
Не общий свободный диалог, а заполнение недостающих слотов.

Пример:
- если нет `event_type` → спросить о типе события;
- если нет `dress_code` и `desired_impression` → спросить один из них;
- если минимальный набор есть → переход к генерации.

## 5.3. После достаточного контекста — только generation
Если у пользователя уже есть достаточно данных, режим не должен снова задавать уточнения или замолкать.

---

# Этап 6. Исправить «Попробовать другой стиль» через анти-повтор

Это ключевой сценарий для разнообразия изображений.

## 6.1. Ввести persistent style history
В session state хранить последние 3–5 style directions:

- palette
- silhouette
- hero garments
- footwear
- accessories
- materials
- styling mood
- composition type
- background family

## 6.2. В decision layer строить `diversity constraints`
При новом `style_exploration` оркестратор должен добавлять ограничения:

- avoid previous palette
- avoid previous hero garments
- avoid previous silhouette family
- avoid previous composition layout
- force material contrast
- force footwear change
- force accessory change

## 6.3. Разделить seed diversity и semantic diversity
Смена random seed недостаточна.

Нужны два уровня разнообразия:

1. **semantic diversity** — другой набор идей, силуэтов, палитры, вещей;
2. **visual diversity** — другой layout, surface, spacing, camera distance.

## 6.4. Передавать previous style directions до самого generation prompt
Нужно проследить, чтобы история предыдущих стилей не терялась между:

- decision layer
- structured brief
- final generation prompt
- Comfy input

---

# Этап 7. Полностью перестроить prompt builder

Сейчас нужен не один длинный описательный prompt, а **структурированный fashion brief**.

## 7.1. Новый формат промежуточного brief
Внутренний объект перед генерацией должен содержать:

- `style_identity`
- `historical_reference`
- `tailoring_logic`
- `color_logic`
- `garment_list`
- `palette`
- `materials`
- `styling_notes`
- `composition_rules`
- `negative_constraints`
- `diversity_constraints`

## 7.2. Prompt builder должен быть двухслойным

### Слой 1. Fashion reasoning
LLM думает как:
- историк моды;
- портной;
- стилист по цвету и силуэту.

### Слой 2. Image prompt compiler
Отдельный builder переводит structured brief в безопасный и стабильный prompt для ComfyUI.

### Почему это важно
Так легче:
- тестировать;
- валидировать;
- заменять модель;
- контролировать однообразие.

---

# Этап 8. Добавить knowledge layer вместо надежды на «умную модель»

Требование «знать лекции Малевича о цвете, историю моды, логику портного» нельзя стабильно решить только ростом размера модели.

Нужен **fashion knowledge layer**.

## 8.1. Структура базы знаний
Создать несколько групп знаний:

- `style_catalog`
- `color_theory`
- `fashion_history`
- `tailoring_principles`
- `materials_fabrics`
- `flatlay_prompt_patterns`

## 8.2. Для каждого стиля хранить
- название;
- краткое описание;
- происхождение;
- ключевые формы;
- характерные вещи;
- палитры;
- материалы;
- аксессуары;
- что нельзя смешивать;
- что делает стиль визуально узнаваемым.

## 8.3. Retrieval перед генерацией ответа
До вызова LLM система должна доставать:
- релевантные карточки стиля;
- правила цвета;
- tailoring notes;
- нужные исторические референсы.

### Результат
Бот будет отвечать не случайной «памятью модели», а опираться на собственные данные проекта.

# Этап 9. Усилить ComfyUI pipeline

Даже отличный prompt будет давать похожие картинки, если workflow слишком жёсткий.

## 9.1. Добавить управляемые visual presets
Нужны разные presets:

- background families
- spacing density
- object count range
- camera distance
- shadow hardness
- layout archetype

## 9.2. Для разных режимов — разные presets
- `garment_matching` — anchor garment centrality high
- `style_exploration` — diversity high
- `occasion_outfit` — practical coherence high

## 9.3. Сохранять метаданные генерации
Для каждого результата хранить:

- final prompt
- negative prompt
- seed
- style id
- visual preset
- palette tags
- garments tags

Это позволит позже анализировать повторы и качество.

# Этап 10. Ввести generation jobs и очереди как обязательный слой

Нельзя держать генерацию изображения как длинный синхронный HTTP-сценарий.

## 10.1. Нужен job queue layer
Например:

- Redis + RQ / Celery
- отдельный worker для image generation
- отдельный worker для style ingestion

## 10.2. Flow
- API route принимает сообщение;
- orchestrator принимает решение;
- если нужна генерация — создаётся job;
- frontend получает `job_id`;
- дальше polling / websocket / SSE;
- когда job завершилась — UI подтягивает результат.

### Почему
Так проект станет устойчивым:
- без таймаутов;
- без двойных кликов;
- без подвисаний запроса.

# Этап 11. Наблюдаемость и диагностика

Сейчас система слишком сложная, чтобы дебажить её «на ощущениях».

## 11.1. Добавить сквозное логирование
Для каждого сообщения писать:

- `session_id`
- `message_id`
- `requested_intent`
- `resolved_intent`
- `decision_type`
- `provider` (`vllm` / fallback)
- `generation_job_id`
- `prompt_hash`
- `style_id`
- `clarification_state`

## 11.2. Метрики
- доля fallback-ответов;
- доля silent failure;
- доля генераций по режимам;
- среднее число уточнений до генерации;
- доля повторяющихся palette/signature items;
- успех парсинга стилей.

# Этап 12. Тестирование сценариев как продукта, а не только кода

## 12.1. Нужны эталонные сценарии
Минимум:

- 5 кейсов `general_advice`
- 5 кейсов `garment_matching`
- 5 кейсов `style_exploration`
- 5 кейсов `occasion_outfit`

## 12.2. Для каждого сценария проверять
- какой режим активирован;
- нужны ли уточнения;
- создалась ли generation job;
- дошёл ли prompt до Comfy;
- отличается ли результат от предыдущего.

## 12.3. Отдельные golden tests на diversity
Например, три последовательных `try another style` не должны повторять:
- доминирующую палитру;
- ключевую обувь;
- главный anchor garment;
- тип силуэта.

# Этап 13. Работа с моделью: когда усиливать, а когда нет

Модель сама по себе не решит архитектурные проблемы.

## Сначала нужно
- разнести режимы;
- добавить state machine;
- починить flow генерации;
- добавить knowledge layer;
- переписать prompt builder;
- ввести diversity constraints.

## После этого уже сравнивать модели
Возможные роли моделей:

- малая / быстрая — классификация, slot extraction, lightweight replies;
- основная reasoning-модель — fashion reasoning и structured brief;
- image generation stack — отдельно.

# Что считать готовым результатом

Проект можно считать приведённым к зрелой архитектуре, если:

1. текстовый чат стабильно отвечает в рамках роли stylist/fashion historian/tailor;
2. `Подобрать к вещи` гарантированно ведёт к генерации после одного-двух уточнений;
3. `Попробовать другой стиль` даёт заметно разные образы подряд;
4. `Что надеть на событие` не теряет контекст после уточнения;
5. база стилей становится рабочим knowledge source, а не просто списком названий;
6. генерация не зависит от длинного синхронного запроса;
7. можно расширять проект новыми режимами без переписывания ядра.

# Практический вывод

Главное, что нужно сделать — это не «подкрутить промпт» и не «поставить модель побольше», а выстроить систему как набор устойчивых слоёв:

- UI command layer
- chat orchestrator
- session state machine
- knowledge retrieval
- structured fashion reasoning
- prompt compiler
- generation jobs
- image workflow presets
- ingestion pipeline для базы стилей

Именно такой подход даст не временно рабочий бот, а стабильный, поддерживаемый и расширяемый продукт.
