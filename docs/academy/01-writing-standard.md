# Academy Writing Standard

Этот файл задаёт единый шаблон для всех будущих учебных глав.

## Шаблон главы

Каждая глава должна содержать разделы в таком порядке:

1. `Что это академически`
2. `Что это простыми словами`
3. `Почему это реализовано именно так`
4. `Какие anti-patterns рядом существуют`
5. `Что реализовано в этом проекте`
6. `Что ещё можно улучшить`
7. `Где это видно в коде`
8. `Разбор на примере`
9. `Команды`
10. `Мини-упражнения`
11. `Что читать дальше`

## Что обязательно включать

- термин;
- академическое определение термина;
- его короткое определение;
- отличие от похожих терминов;
- пример из реальной жизни;
- пример из этого проекта;
- если прямого примера в проекте нет, то обязательный простой вымышленный пример;
- объяснение, почему текущее решение выбрано именно так;
- блок с возможными улучшениями на стороне backend/frontend;
- список файлов;
- конкретные функции/классы/роуты;
- хотя бы одну практическую команду;
- хотя бы одно маленькое упражнение.

## Как оформлять сравнения

Если термин лучше понимается через сравнение, обязательно добавлять блок:

- `Чем X отличается от Y`

Например:
- JavaScript vs TypeScript
- Postgres vs MongoDB
- Redis vs Postgres
- Elasticsearch vs SQL search
- React vs vanilla JavaScript
- migration vs seed
- model vs schema vs DTO

## Как оформлять примеры из проекта

Для каждого понятия по возможности указывать:
- 1 файл-вход;
- 1 файл с логикой;
- 1 файл, где виден результат использования.

Если понятие напрямую не реализовано в проекте, нужно:
- явно написать, что это comparative material;
- дать маленький вымышленный пример;
- отдельно пояснить, как этот термин выглядел бы в нашем проекте, если бы мы его внедряли.

Пример:
- термин: `API client`
- вход: `apps/frontend/src/shared/api/client.ts`
- использование: `apps/frontend/src/features/chat/model/useStylistChatSimple.ts`
- эффект: `apps/frontend/src/features/chat/ui/ChatWindowSimpleSurface.tsx`

## Уровни объяснения

Нужны сразу три уровня:

### Уровень 1. Для совсем новичка

Объяснение без жаргона.

### Уровень 2. Для начинающего разработчика

Объяснение уже с правильными терминами.

### Уровень 3. Для чтения реального кода

Как это выглядит в конкретных файлах проекта.

## Что не делать

- не прыгать сразу в код без определения;
- не объяснять термин через ещё пять незнакомых терминов;
- не писать только теорию без привязки к проекту;
- не писать только список файлов без объяснения идей.
