# Жёсткое требование на исправление ingestion-пайплайна

## Статус решения

Текущее поведение ingestion-пайплайна считается архитектурно неприемлемым и подлежит исправлению.

Фактические симптомы:
- пайплайн делает обращения к `https://aesthetics.fandom.com/robots.txt`;
- `robots.txt` возвращает `HTTP 403`;
- из-за этого ingestion деградирует или блокируется;
- worker не даёт нормальной диагностической картины;
- при зависании/ожидании оператор не понимает, что именно происходит;
- логирование недостаточно для поддержки и эксплуатации.

Для данного проекта это недопустимо.

---

## Архитектурное решение верхнего уровня

### Обязательное требование

**Полностью убрать использование `robots.txt` из ingestion pipeline.**

Не ослабить.
Не сделать optional.
Не оставить как advisory.
Не оборачивать в feature flag по умолчанию.
Не сохранять старую ветку поведения “на всякий случай”.

Именно:
- исключить `robots.txt` из runtime path;
- исключить `robots.txt` из preflight checks;
- исключить `robots.txt` из access gating;
- исключить `robots.txt` из source availability decision.

### Обоснование

В текущем продукте используется **один донор**, и решение по доступности этого донора должно приниматься только по факту доступности его целевого API.

Следовательно:
- `robots.txt` не несёт полезной операционной ценности;
- `robots.txt` добавляет ложную точку отказа;
- `robots.txt` ухудшает надёжность ingestion без пользы для системы;
- `robots.txt` загрязняет transport path инфраструктурной логикой, не связанной с доменной задачей.

---

## Доменные правила после исправления

### Единственный критерий доступности донора

Источник считается доступным, если:
- его API endpoint доступен;
- API возвращает корректный ответ в ожидаемом формате.

Источник считается недоступным, если:
- API недоступен по сети;
- API возвращает устойчивую ошибку доступа;
- API возвращает невалидный payload для заявленного fetch mode.

### Недопустимое поведение

Недопустимо, чтобы решение “можно ли загружать данные из донора” принималось на основании:
- `robots.txt`;
- HTML preflight;
- сторонних advisory endpoint;
- скрытых инфраструктурных эвристик вне доменной модели.

---

## Требование к архитектуре

Исправление должно соответствовать:
- OOP;
- SOLID;
- clean architecture;
- низкой связности;
- высокой сопровождаемости;
- отсутствию регресса в уже работающих частях ingestion pipeline.

---

## Обязательные требования к реализации

### 1. Полностью удалить robots-логику из runtime path

Необходимо удалить из рабочего пути исполнения:
- загрузку `robots.txt`;
- парсинг robots policy;
- проверки вида `_ensure_allowed_by_robots`;
- любые wait/gate/deny решения, основанные на результате robots-проверки;
- любые исключения, связанные с robots access.

Если существует внутренний helper/transport hook, связанный с robots:
- он должен быть удалён из основного execution flow;
- код должен быть либо полностью удалён, либо переведён в неиспользуемый legacy слой, не подключённый к runtime.

### 2. Оставить единую точку принятия решения по доступности

В ingestion должно остаться ровно одно основание для проверки источника:
- успешность API-запроса к целевому donor API.

Нельзя допускать вторую скрытую проверку доступа в другом месте.

### 3. Не размазывать условности по коду

Запрещено:
- вставлять хаки в стиле `if robots fails: continue` в разных местах;
- оставлять полуживые вызовы robots и просто гасить исключения;
- делать точечные hotfix-и в нескольких слоях одновременно.

Нужно привести код к состоянию, где логика доступа к источнику проста и однозначна:

> ingestion работает через API, точка.

### 4. Не ломать существующую модель pipeline

Нельзя нарушить:
- enqueue jobs;
- discovery jobs;
- detail jobs;
- worker execution;
- state transitions;
- persistence результатов;
- already working admin endpoints.

Удаление robots не должно привести к регрессии очередей, сохранения, нормализации или отчётности.

---

## Требование к логированию

Текущее логирование признано недостаточным.

Нужно добавить **нормальное структурированное прикладное логирование** для ingestion pipeline.

### Логирование обязательно должно покрывать:

#### 1. Запуск команды
- какой mode запущен;
- source_name;
- limit/max-jobs;
- worker options.

#### 2. Работа с job queue
- job claimed;
- job id;
- job type;
- job started;
- job finished;
- job failed;
- retry/cooldown state;
- enqueue detail jobs result.

#### 3. Работа с donor API
- API request started;
- endpoint/method/logical operation;
- API request succeeded;
- API request failed;
- статус ответа;
- причина отказа;
- длительность запроса.

#### 4. Discovery stage
- discovered_count;
- selected_count;
- enqueued_count;
- reused_count.

#### 5. Detail processing stage
- какой source page обрабатывается;
- какой style title/slug извлечён;
- сохранён ли style;
- created/updated/skipped/error.

#### 6. Worker waiting behaviour
Если worker уходит в ожидание:
- это должно быть залогировано явно;
- должна быть указана причина ожидания;
- должна быть указана длительность ожидания;
- оператор должен понимать, это idle, cooldown, throttle или блокировка по состоянию.

#### 7. Итог операции
- processed jobs count;
- failed jobs count;
- created/updated styles count;
- stop reason.

---

## Требование к стилю логирования

Логирование должно быть:
- машинно-читаемым или хотя бы строго структурированным;
- единообразным;
- без шумовых бессмысленных сообщений;
- ориентированным на поддержку и расследование инцидентов.

Недостаточно просто добавить `print()`.

Предпочтительно:
- единый logger ingestion domain;
- message templates с контекстом;
- стабильные ключи полей.

---

## Требование по OOP / SOLID

### Single Responsibility Principle
- transport отвечает только за HTTP;
- pipeline orchestrates jobs;
- scraper/fetcher отвечает за извлечение данных;
- logging concern оформлен централизованно и не размазан хаотично.

### Open/Closed Principle
- после удаления robots логика доступа должна оставаться расширяемой через нормальные API/fetch abstractions;
- но без сохранения robots как скрытой обязательной зависимости.

### Liskov Substitution Principle
- любые fetcher-ы, работающие через API, должны использовать единый контракт доступа и логирования.

### Interface Segregation Principle
- компоненты, не занимающиеся логированием или HTTP, не должны зависеть от деталей их реализации.

### Dependency Inversion Principle
- orchestration слой зависит от абстракций fetch/report/logging, а не от низкоуровневых деталей реализации доступа.

---

## Требование к чистоте архитектуры

Необходимо, чтобы после исправления архитектурная схема выглядела так:

- `job_pipeline` управляет очередью и стадиями;
- `batch_runner` координирует discovery/detail processing;
- `scraper/fetcher` обращается напрямую к donor API;
- `transport` делает HTTP;
- `logging` отражает жизненный цикл пайплайна;
- `robots.txt` нигде не участвует.

Именно полное устранение лишнего слоя, а не косметическая маскировка проблемы.

---

## Требование к обратной совместимости

Нужно не сломать:
- текущий workflow enqueue → worker → persistence;
- текущие успешные API-based fetch сценарии;
- UI/admin overview endpoints;
- структуры job/result/report, если они уже используются интерфейсом.

Если какие-то поля исторически существовали из-за robots logic и больше не нужны:
- допускается их удалить только если это не ломает текущие контракты;
- если удаление рискованно — сохранить интерфейс, но убрать влияние этих полей на runtime.

---

## Требование к тестированию

Обязательно покрыть тестами:

1. Discovery через donor API без `robots.txt`.
2. Успешную постановку detail jobs.
3. Успешную обработку detail job.
4. Worker logging на стадиях claim/start/success/fail/wait.
5. Ошибку donor API как доменно понятную ошибку.
6. Отсутствие любых вызовов `robots.txt` в runtime path.

Нужно гарантировать, что повторно подобный регресс не вернётся скрыто.

---

## Запрещённые подходы

Запрещено:
- частично комментировать один вызов и оставлять два других;
- silently ignore `403` от robots;
- оставлять скрытый fallback на robots;
- разбрасывать правки по коду без единой архитектурной точки ответственности;
- ограничиться shell workaround без исправления Python-кода.

---

## Ожидаемый результат

После исправления система должна работать так:

1. enqueue jobs создаёт discovery job;
2. worker обрабатывает discovery job через donor API;
3. detail jobs создаются и выполняются;
4. styles сохраняются;
5. в логах видно, что происходит;
6. `robots.txt` нигде не вызывается;
7. `403 robots.txt` больше никак не влияет на систему.

---

## Финальная формулировка задачи

Требуется полностью удалить зависимость ingestion pipeline от `robots.txt` и перевести весь механизм доступа к единственному источнику данных на модель, в которой решение о доступности принимается исключительно по donor API. Параллельно требуется внедрить полноценное структурированное логирование жизненного цикла ingestion jobs и worker execution. Реализация должна быть выполнена как чистое архитектурное исправление, соответствующее OOP, SOLID и clean architecture, без временных хаков и без регресса в уже работающих частях системы.
