# Что Такое Процесс, Память И Операционная Система

Это следующая фундаментальная глава курса.

После неё слова `процесс`, `память`, `операционная система`, `порт`, `сокет`, `переменные окружения` и `права доступа` уже не должны звучать как что-то мистическое. Именно на них дальше будут опираться темы про Docker, backend, базы данных, сеть, vLLM и ComfyUI.

## Что Это Академически

### Процесс

Процесс — это экземпляр выполняемой программы, которому операционная система выделила собственное пространство памяти, идентификатор, набор ресурсов и контекст выполнения.

У процесса обычно есть:
- код программы;
- данные;
- память;
- открытые файлы;
- сетевые соединения;
- переменные окружения;
- права доступа;
- идентификатор процесса, или PID.

### Поток

Поток — это единица выполнения внутри процесса.

Если говорить строго, процесс может содержать один или несколько потоков, которые работают в одном адресном пространстве памяти процесса.

### Память

Оперативная память — это ресурс, в котором процесс во время работы хранит:
- загруженный код;
- переменные;
- временные данные;
- буферы ввода-вывода;
- сетевые данные;
- внутреннее состояние библиотек и рантайма.

### Операционная Система

Операционная система — это программная система, которая управляет аппаратными ресурсами компьютера и предоставляет программам стандартные механизмы работы с процессами, памятью, файлами, сетью, правами доступа и устройствами.

Упрощённо:
- программа не разговаривает с железом напрямую;
- между ними почти всегда стоит ОС;
- именно ОС решает, какой процесс когда выполняется, сколько памяти ему можно дать и какой порт он может открыть.

### Дескриптор И Handle

Дескриптор файла в Unix-подобных системах и handle в Windows — это идентификатор ресурса, который процесс получил от ОС.

Это может быть:
- файл;
- сокет;
- канал;
- устройство;
- стандартный ввод, вывод и поток ошибок.

То есть процесс работает не “с файлом вообще”, а с конкретным открытым ресурсом, который ОС ему выдала.

### Сокет И Порт

Сокет — это программная конечная точка сетевого взаимодействия.

Порт — это числовой адрес внутри сетевого хоста, на котором процесс может слушать входящие соединения или через который он может инициировать исходящее соединение.

Например:
- backend слушает `8000`;
- frontend слушает `3000`;
- ComfyUI слушает `8188`;
- vLLM отвечает на `8001`.

### Переменные Окружения

Переменные окружения — это набор именованных значений, который ОС передаёт процессу при запуске.

Через них часто задают:
- URL базы данных;
- порты;
- ключи доступа;
- режим окружения;
- фиче-флаги;
- пути к данным и логам.

### Права Доступа

Права доступа — это правила, определяющие, что процессу разрешено делать:
- читать файл;
- писать файл;
- открывать порт;
- подключаться к сокету;
- запускать другой процесс;
- читать каталог;
- использовать сетевой ресурс.

Если прав не хватает, код может быть абсолютно правильным, но программа всё равно не заработает.

## Что Это Простыми Словами

Если совсем по-человечески:

- программа — это рецепт;
- процесс — это уже реально приготовляемое блюдо;
- память — это рабочий стол, где лежат ингредиенты и промежуточные результаты;
- ОС — это кухня, администратор, правила безопасности и расписание, которые позволяют всему происходить;
- порт — это номер двери;
- сокет — это сама дверь для разговора по сети;
- переменные окружения — это бумажка с настройками, которую повар получает перед началом смены.

Именно поэтому один и тот же код может вести себя по-разному:
- на другой машине;
- под другим пользователем;
- с другим `.env`;
- с другим количеством памяти;
- с занятым или свободным портом;
- под Windows, Linux или внутри контейнера.

## Почему Это Реализовано Именно Так

В реальном проекте почти никогда не существует “одна магическая программа, которая делает всё”.

Вместо этого система состоит из нескольких процессов, за которые отвечает ОС:

- процесс frontend;
- процесс backend;
- процесс PostgreSQL;
- процесс Redis;
- процесс Elasticsearch;
- процесс vLLM;
- процесс ComfyUI;
- дополнительные служебные процессы контейнеров, прокси, логирования и т.д.

Это сделано не случайно.

Разделение по процессам нужно, чтобы:
- изолировать ответственность;
- не смешивать UI, бизнес-логику и хранение данных;
- перезапускать одну часть системы без полного падения всех остальных;
- распределять память и CPU между компонентами;
- выносить тяжёлые части, например AI runtime, отдельно от web-приложения;
- масштабировать части системы независимо друг от друга.

## Какие Anti-Patterns Рядом Существуют

### Anti-pattern 1. Думать, что файл и процесс — одно и то же

Это неверно.

Например:
- `app/main.py` — это файл;
- `uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload` — это команда запуска;
- уже запущенный backend — это процесс.

Файл лежит на диске. Процесс живёт в памяти и выполняется ОС.

### Anti-pattern 2. Путать процесс, поток и async-задачу

Это три разные вещи:
- процесс — сущность уровня ОС;
- поток — сущность внутри процесса;
- async-задача — сущность уровня рантайма и event loop.

В нашем backend poller генераций создаётся как async-задача, а не как отдельный процесс: `apps/backend/app/main.py:28`.

### Anti-pattern 3. Думать, что Docker отменяет ОС

Docker не “убирает” операционную систему.

Контейнеры всё равно используют:
- ядро ОС хоста;
- процессы;
- память;
- файловую систему;
- сеть;
- порты.

Контейнер — это не магия, а управляемое окружение для процессов.

### Anti-pattern 4. Игнорировать переменные окружения

Очень частая ошибка новичка: менять код там, где нужно менять `.env`.

В этом проекте через переменные окружения задаются:
- база данных;
- Redis;
- Elasticsearch;
- CORS;
- адреса vLLM и ComfyUI;
- таймауты и поведение генерации.

Это видно в `apps/backend/app/core/config.py:11, 17-20, 31, 33, 48, 80`.

### Anti-pattern 5. Считать, что “если код написан, значит программа должна запускаться”

Нет.

Программа может не запуститься, если:
- порт уже занят;
- не хватает памяти;
- нет доступа к файлу;
- нет переменной окружения;
- не открыта сеть;
- контейнер не видит другой сервис;
- выбран неверный Python или неверный runtime.

В реальном проекте это происходит постоянно.

## Что Реализовано В Этом Проекте

### 1. Backend — отдельный процесс веб-сервера

В `docker-compose.yml:91` backend запускается командой:

```text
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

Это означает:
- ОС создаёт процесс `uvicorn`;
- процесс слушает порт `8000`;
- внутрь этого процесса загружается приложение `FastAPI`.

В `apps/backend/app/main.py:40` создаётся объект `FastAPI`, а в `apps/backend/app/main.py:56` к нему подключается роутер.

То есть:
- файл `main.py` — это код;
- процесс `uvicorn` — это уже работающий сервер;
- порт `8000` — это точка входа по сети.

### 2. У backend есть собственное состояние в памяти

В `apps/backend/app/main.py:21-32` backend использует lifespan и создаёт фоновую async-задачу poller генераций:

- `apps/backend/app/main.py:21` — стартует lifecycle;
- `apps/backend/app/main.py:25` — создаётся каталог media;
- `apps/backend/app/main.py:28` — создаётся async-задача poller;
- `apps/backend/app/main.py:32` — задача корректно отменяется при остановке процесса.

Это хороший пример того, что процесс — это не “одна функция”, а живущее состояние:
- у него есть память;
- у него есть фоновые задачи;
- у него есть старт и завершение.

### 3. Процесс backend получает настройки от ОС через environment

В `apps/backend/app/core/config.py` класс `Settings` читает переменные окружения:

- `apps/backend/app/core/config.py:11` — `.env` подключается как источник конфигурации;
- `apps/backend/app/core/config.py:17-20` — читаются URL базы, Redis и Elasticsearch;
- `apps/backend/app/core/config.py:31` — читаются CORS origins;
- `apps/backend/app/core/config.py:33` — читается адрес ComfyUI;
- `apps/backend/app/core/config.py:48` — читается адрес vLLM;
- `apps/backend/app/core/config.py:80` — создаётся объект настроек.

Это значит, что один и тот же Python-код можно запускать:
- локально;
- в контейнере;
- на другом сервере;
- в staging;
- в production,

а поведение будет меняться не из-за переписывания кода, а из-за другого окружения процесса.

### 4. У backend есть сетевые соединения к другим процессам

В `apps/backend/app/db/session.py` backend создаёт async engine и сессии базы данных:

- `apps/backend/app/db/session.py:10` — создаётся engine;
- `apps/backend/app/db/session.py:11` — создаётся фабрика сессий;
- `apps/backend/app/db/session.py:14-15` — выдаётся рабочая DB session.

Это важно понимать правильно:
- Postgres — это другой процесс;
- backend не “лезет в таблицу напрямую”;
- backend открывает сетевое соединение к внешнему сервису базы данных.

То же мышление дальше переносится и на Redis, Elasticsearch, vLLM и ComfyUI.

### 5. Frontend — это тоже отдельный процесс и отдельный сетевой клиент

В `docker-compose.yml:103` frontend получает внутренний адрес backend:

```text
INTERNAL_API_URL=http://backend:8000/api/v1
```

А в `apps/frontend/src/shared/api/client.ts` уже видно, как этот процесс строит запросы:

- `apps/frontend/src/shared/api/client.ts:23` — собирается URL;
- `apps/frontend/src/shared/api/client.ts:47` — выполняется `fetch`;
- `apps/frontend/src/shared/api/client.ts:54` — обрабатываются ошибки ответа;
- `apps/frontend/src/shared/api/client.ts:259` — создаётся `FormData` для загрузки файлов;
- `apps/frontend/src/shared/api/client.ts:275` — отправляется сообщение стилисту.

То есть frontend — это не “картинка сайта”, а отдельный процесс, который:
- держит память;
- знает адрес backend;
- инициирует сетевые соединения;
- отправляет JSON и multipart-данные.

### 6. Docker Compose раскладывает процессы по сервисам и портам

В `docker-compose.yml` видно, какие процессы вообще составляют среду:

- `docker-compose.yml:3` — Postgres;
- `docker-compose.yml:21` — Redis;
- `docker-compose.yml:35` — Elasticsearch;
- `docker-compose.yml:52-91` — backend;
- `docker-compose.yml:93-109` — frontend.

Именно там же фиксируются порты:
- `docker-compose.yml:85` — backend наружу на `8000`;
- `docker-compose.yml:109` — frontend наружу на `3000`;
- `docker-compose.yml:47` — healthcheck Elasticsearch через `localhost:9200`.

Это очень важная идея:

процессы не просто “живут рядом”, а получают от ОС и от контейнерной сети конкретные адреса, имена и порты.

### 7. В проекте есть процессы вне Docker

Не всё живёт внутри `docker-compose`.

По runtime-докам проекта:
- `vLLM` работает на Windows host и доступен по `http://192.168.50.141:8001/v1` — это зафиксировано в `docs/07-vllm-runtime-reference.md`;
- `ComfyUI` работает на Windows host и доступен по `http://192.168.50.141:8188` — это зафиксировано в `docs/08-comfyui-runtime-reference.md`;
- автозапуск этих компонентов описан в `docs/06-autostart-map.md`.

Это отличный инженерный пример:
- одна система может состоять из процессов не только внутри одного контейнера и даже не только внутри одной ОС;
- часть процессов может жить в Docker;
- часть — в WSL;
- часть — напрямую на Windows host.

Но для frontend и backend всё равно остаётся одна и та же базовая модель: есть другой процесс, есть сетевой адрес, есть порт, есть протокол общения.

### 8. ОС управляет правами, путями и доступом к ресурсам

Даже такие обычные действия, как:
- создать каталог для media;
- открыть лог-файл;
- запустить Python из `venv`;
- прочитать `.env`;
- открыть порт `8000` или `8188`,

возможны только потому, что ОС разрешила процессу сделать это.

Это хорошо видно в runtime-доках:
- `docs/06-autostart-map.md`
- `docs/07-vllm-runtime-reference.md`
- `docs/08-comfyui-runtime-reference.md`

Именно поэтому ошибки уровня ОС часто выглядят так:
- порт уже занят;
- файл не найден;
- доступ запрещён;
- сервис не поднялся;
- соединение сброшено;
- процесс убит по памяти или таймауту.

## Разбор На Примере

### Простой Вымышленный Пример

Представим маленькую программу `notes-server`, которая слушает порт `9000` и хранит заметки в памяти.

Когда ОС запускает её:
- создаётся процесс;
- процесс получает память;
- процесс открывает сокет на `9000`;
- процесс получает переменные окружения, например `APP_ENV=dev`;
- процесс начинает принимать запросы.

Если:
- порт `9000` уже занят,

то программа не сможет стартовать, даже если код у неё правильный.

Если:
- у процесса нет права писать в каталог `logs`,

то логирование упадёт, даже если логика приложения написана без ошибок.

### Пример Из Текущего Проекта

Когда пользователь открывает сайт и пишет сообщение стилисту, происходит цепочка процессов:

1. Процесс frontend отвечает в браузере на `3000`.
2. Frontend строит HTTP-запрос к backend.
3. Процесс backend слушает `8000` и принимает запрос.
4. Backend может открыть соединение к Postgres, Redis и Elasticsearch.
5. Backend может отправить запрос к vLLM на `8001`.
6. Backend может отправить запрос к ComfyUI на `8188`.
7. Ответ возвращается обратно в frontend.

Во всей этой цепочке ОС участвует на каждом шаге:
- выделяет память;
- держит процессы;
- открывает порты;
- поддерживает сокеты;
- управляет сетью;
- хранит файлы;
- проверяет доступ.

## Команды

### Посмотреть контейнерные процессы проекта

```bash
docker compose ps
docker compose top
```

### Посмотреть, какие порты опубликованы наружу

```bash
docker compose ps
netstat -ano | findstr :8000
netstat -ano | findstr :3000
```

```bash
docker compose exec backend printenv
```

### Посмотреть, что backend вообще жив

```bash
curl http://127.0.0.1:8000/health
```

```powershell
Get-Process | Where-Object { $_.ProcessName -like "*python*" -or $_.ProcessName -like "*wsl*" -or $_.ProcessName -like "*node*" }
```

```powershell
netstat -ano | findstr :8188
```
