# Что Такое Файл, Каталог И Проект

Это вторая фундаментальная глава курса.

После неё слова `файл`, `папка`, `путь`, `корень проекта`, `исходники`, `конфиг`, `репозиторий` и `дерево проекта` уже не должны быть абстракцией.

## Что Это Академически

### Файл

Файл — это именованная единица хранения данных в файловой системе.

Файл может содержать:
- текст;
- двоичные данные;
- конфигурацию;
- исходный код;
- изображения;
- шрифты;
- логи;
- архивы.

### Каталог

Каталог, или директория, — это структурная сущность файловой системы, которая хранит ссылки на файлы и другие каталоги и образует иерархию хранения.

Проще говоря:
- файл хранит содержимое;
- каталог хранит организацию.

### Проект

Программный проект — это согласованный набор файлов, каталогов, конфигурации, зависимостей и соглашений, вместе образующих одну разрабатываемую систему.

То есть проект — это не просто папка.

Проект обычно включает:
- исходный код;
- конфигурацию;
- инструкции запуска;
- описание зависимостей;
- документацию;
- иногда миграции, тесты, media и скрипты.

## Что Это Простыми Словами

Если совсем по-человечески:

- файл — это один конкретный предмет с содержимым;
- папка — это место, где такие предметы разложены по категориям;
- проект — это уже целая мастерская, где у каждого файла и каждой папки есть роль.

Например:
- `README.md` — это текстовый файл с описанием проекта;
- `apps/frontend` — это папка с клиентской частью;
- `c:\dev\portfolio` — это корень проекта;
- весь репозиторий `portfolio` — это и есть проект.

## Почему Это Реализовано Именно Так

В реальной разработке структура проекта нужна не "для красоты", а чтобы:

- быстро находить код;
- понимать границы ответственности;
- не смешивать frontend, backend, docs и media в одну кучу;
- проще разворачивать и сопровождать систему;
- передавать проект другим разработчикам без устных объяснений.

Именно поэтому в этом проекте уже есть верхнеуровневое разделение:

- `apps` — прикладной код;
- `docs` — документация;
- `media` — пользовательские или служебные медиа-артефакты;
- `.env.example` — шаблон переменных окружения;
- `docker-compose.yml` — точка сборки и запуска окружения;
- `README.md` — входная документация по репозиторию.

Это уже не случайный набор файлов, а архитектурно организованное дерево.

## Какие Anti-Patterns Рядом Существуют

### Anti-pattern 1. Складывать всё в одну папку

Например:
- код фронта;
- код бэка;
- картинки;
- логи;
- SQL;
- markdown-документы;
- временные файлы.

Почему это плохо:
- проект становится трудно читать;
- растёт риск случайных ошибок;
- новому разработчику тяжелее понять, где что лежит.

### Anti-pattern 2. Давать файлам имена без роли

Например:
- `test2.py`
- `newfile.ts`
- `final_final_v2.md`

Почему это плохо:
- имя файла не объясняет назначение;
- структура не помогает читать систему;
- поиск по проекту становится менее полезным.

### Anti-pattern 3. Не различать конфиг, код и данные

Например, когда рядом лежат:
- runtime-конфиги;
- пользовательские загрузки;
- исходники приложения;
- локальные временные файлы.

Почему это плохо:
- можно случайно закоммитить не то;
- сложнее автоматизировать сборку;
- труднее понять, что является частью программы, а что артефактом запуска.

## Что Реализовано В Этом Проекте

На верхнем уровне проекта уже видна разумная структура.

### Корень проекта

Корень здесь:
- `c:\dev\portfolio`

Именно отсюда:
- запускается `docker compose`;
- читается `.env`;
- начинается навигация по репозиторию;
- работают относительные пути во многих командах.

### Верхнеуровневые каталоги

На верхнем уровне лежат:

- `apps`
- `docs`
- `media`

Что это означает:

- `apps` — основной прикладной код;
- `docs` — документация по проекту и учебный контур;
- `media` — медиа-артефакты;
- корневые конфиги лежат рядом с ними, потому что относятся ко всему репозиторию.

### Каталог apps

Внутри `apps` уже видно разделение по прикладным частям системы:

- `apps/frontend`
- `apps/backend`

Это очень важная идея:
- проект один;
- приложений внутри него несколько.

### Каталог docs

`docs` — это не мусорная папка "со всякими текстами", а отдельный слой знаний о проекте.

Там лежат:
- runtime-доки;
- roadmap;
- setup-checklist;
- academy.

То есть документация у проекта тоже структурирована.

## Что Ещё Можно Улучшить

Даже при хорошей текущей структуре со временем почти всегда появляется пространство для улучшений.

Например:

- при росте проекта можно ещё жёстче фиксировать правила именования файлов и папок;
- можно отдельно документировать соглашения: где должны лежать тесты, скрипты, миграции, assets;
- можно ввести дополнительные checks против случайного попадания временных файлов в репозиторий;
- можно отдельно объяснить, какие папки являются source of truth, а какие считаются runtime-output.

## Разбор На Примере

### Простой Вымышленный Пример

Представим маленький учебный проект `weather-app`.

Его дерево могло бы выглядеть так:

```text
weather-app/
  README.md
  package.json
  src/
    index.ts
    api.ts
    ui.ts
  public/
    logo.png
```

Как это читать:

- `weather-app` — это проект;
- `README.md` — файл с описанием;
- `src` — каталог исходников;
- `index.ts` — файл входа;
- `public` — каталог статических файлов.

### Пример Из Текущего Проекта

У нас дерево на верхнем уровне уже устроено примерно так:

```text
portfolio/
  apps/
    backend/
    frontend/
  docs/
    academy/
  media/
  .env.example
  .gitattributes
  .gitignore
  docker-compose.yml
  README.md
```

Из этой схемы сразу можно сделать выводы:

- проект не монолитный в смысле одной папки `src`, а multi-part;
- `frontend` и `backend` живут отдельно;
- documentation считается полноценной частью проекта;
- корневые конфиги управляют всем репозиторием сразу.

## Абсолютный И Относительный Путь

Это один из самых важных практических моментов.

### Абсолютный путь

Абсолютный путь начинается от корня файловой системы.

Пример:

```text
c:\dev\portfolio\apps\frontend\package.json
```

Он хорош тем, что однозначен.

### Относительный путь

Относительный путь считается от текущей рабочей папки.

Если вы уже находитесь в:

```text
c:\dev\portfolio
```

то тот же файл можно указать так:

```text
apps\frontend\package.json
```

Это важно, потому что большинство команд в терминале работает именно относительно текущей директории.

## Расширение Файла

Расширение — это часть имени после точки, которая обычно помогает понять тип файла.

Примеры из проекта:

- `.md` — Markdown-документация
- `.py` — Python-файл
- `.ts` — TypeScript
- `.tsx` — TypeScript + JSX
- `.json` — структурированные данные
- `.yml` — конфигурация
- `.mjs` — JavaScript-модуль

Важно:
- расширение полезно, но не "магически определяет смысл";
- смысл файла задаётся не только расширением, но и местом в структуре проекта.

Например:
- `README.md` и `docs/academy/README.md` оба имеют расширение `.md`, но роли у них разные;
- `package.json` и `tsconfig.json` оба `.json`, но первый описывает npm-пакет, а второй — конфиг TypeScript.

## Команды

### Посмотреть верхний уровень проекта

```bash
Get-ChildItem
```

### Посмотреть только имена верхнеуровневых папок

```bash
Get-ChildItem -Name
```

### Посмотреть содержимое каталога apps

```bash
Get-ChildItem apps
```

### Рекурсивно найти все markdown-файлы

```bash
rg --files docs | rg "\.md$"
```

### Найти все package/config-файлы в проекте

```bash
rg --files | rg "package\.json|tsconfig\.json|docker-compose\.yml|\.env\.example|README\.md"
```
