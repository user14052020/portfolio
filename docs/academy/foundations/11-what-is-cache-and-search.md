# Что Такое Кэш И Поиск

Это следующий фундаментальный шаг курса.

После этой главы слова `cache`, `Redis`, `TTL`, `search index`, `full-text search` и `Elasticsearch` должны перестать смешиваться в одну абстрактную массу.

Именно здесь становится понятно:
- почему база данных и кэш не одно и то же;
- почему поиск по тексту часто живёт отдельно от основной базы;
- зачем в одном проекте одновременно могут жить `PostgreSQL`, `Redis` и `Elasticsearch`;
- как именно эти роли разделены в `portfolio`.

## Что Это Академически

### Кэш

Кэш — это промежуточное хранилище данных, используемое для ускорения повторного доступа к уже вычисленным, недавно полученным или временно важным значениям.

Ключевая идея кэша:
- не хранить систему истины;
- а сокращать время доступа и уменьшать нагрузку на более дорогой слой.

### Redis

Redis — это in-memory система хранения структур данных формата key-value, ориентированная на очень быстрые операции чтения, записи и работы с временными состояниями.

Redis умеет:
- хранить строки, хэши, множества и списки;
- задавать TTL, то есть время жизни значения;
- использоваться как кэш;
- использоваться как очередь;
- использоваться как координатор блокировок и временных флагов.

### Поисковый Индекс

Поисковый индекс — это специальная структура данных, оптимизированная для быстрого поиска по содержимому документов, а не только по точному совпадению полей.

Такой индекс обычно строится отдельно от основной транзакционной базы.

### Full-Text Search

Full-text search, или полнотекстовый поиск, — это подход, при котором система ищет не только точные значения полей, но и слова, фразы, формы слов и релевантные совпадения внутри текста.

### Elasticsearch

Elasticsearch — это поисковый и аналитический движок, который хранит документы в индексах и оптимизирован для полнотекстового поиска, фильтрации, ранжирования и агрегаций.

Для нас важно, что Elasticsearch:
- хорошо подходит для поиска по контенту;
- хранит отдельный поисковый слепок данных;
- не обязан быть системой истины;
- обычно работает рядом с основной базой, а не вместо неё.

## Что Это Простыми Словами

Если по-человечески:

- `PostgreSQL` говорит: "я храню настоящие данные проекта, которые нельзя терять и ломать";
- `Redis` говорит: "я очень быстро держу временные состояния, блокировки и короткоживущие значения";
- `Elasticsearch` говорит: "я умею быстро искать по словам и текстовым полям лучше, чем обычный SQL-фильтр".

Полезная грубая модель для первого приближения:

- база данных = официальный архив и источник истины;
- кэш = быстрый блокнот рядом с архивом;
- поисковый индекс = специально подготовленный каталог для поиска по содержимому.

## Чем Кэш Отличается От Базы Данных

База данных обычно нужна для:
- надёжного долгого хранения;
- связей между сущностями;
- транзакций;
- консистентности.

Кэш обычно нужен для:
- очень быстрого доступа;
- временных значений;
- повторного использования уже вычисленного результата;
- координации процессов.

Главная разница:
- если Redis внезапно очистится, система должна уметь восстановиться;
- если PostgreSQL очистится, это уже потеря основных данных проекта.

Именно поэтому кэш нельзя путать с системой истины.

## Чем Полнотекстовый Поиск Отличается От Обычного SQL-Запроса

Обычный SQL-запрос хорошо подходит, когда мы точно знаем:
- по какому полю фильтруем;
- какое значение ищем;
- как сортируем результат.

Примеры хороших SQL-вопросов:
- "дай все опубликованные проекты";
- "найди job по public_id";
- "выведи все blog posts из категории news".

Полнотекстовый поиск нужен, когда пользователь мыслит словами, а не строгими полями:
- "найди статьи про стиль и ткани";
- "покажи проекты, где есть FastAPI и Redis";
- "поиск по заголовкам и описаниям сразу".

Здесь Elasticsearch часто удобнее, потому что он:
- индексирует текст специально под поиск;
- умеет multi-match по нескольким полям;
- умеет релевантность.

## Простые Вымышленные Примеры

### Пример 1. Кэш

Представим сайт погоды.

Если тысяча пользователей подряд запрашивает погоду в Екатеринбурге, сервер может:
- один раз получить данные из внешнего API;
- сохранить ответ в Redis на 60 секунд;
- следующие запросы читать из Redis.

Тогда внешний сервис дёргается реже, а пользователи получают ответ быстрее.

### Пример 2. Поиск

Представим библиотеку книг.

Если мы хотим найти книгу по точному `id`, обычной базы достаточно.

Но если читатель пишет:
- "книга про море и путешествие";
- "что-нибудь про архитектуру древнего Рима";

то удобнее уже поисковый индекс, который умеет искать по текстовому содержимому, а не только по точным значениям колонок.

## Что Реализовано В Этом Проекте

## 1. Redis И Elasticsearch Поднимаются Как Отдельные Сервисы

Это видно в [`docker-compose.yml`](../../../docker-compose.yml#L20):
- сервис `redis` описан в строках `20-32`;
- сервис `elasticsearch` описан в строках `34-50`;
- backend получает `REDIS_URL` и `ELASTICSEARCH_URL` через переменные окружения в строках `60-76`.

Это важный архитектурный сигнал:
- кэш и поиск живут отдельно;
- backend только подключается к ним;
- они не встроены внутрь Python-процесса как случайные глобальные словари.

## 2. Конфиг Backend Явно Разделяет Базу, Redis И Elasticsearch

Это видно в [`config.py`](../../../apps/backend/app/core/config.py#L17):
- `database_url` и `sync_database_url` живут в строках `17-18`;
- `redis_url` и `elasticsearch_url` живут в строках `19-20`;
- `enable_search_indexing` включается отдельным флагом в строке `62`.

Это полезный паттерн:
- база данных отвечает за основное хранение;
- Redis отвечает за быстрые временные состояния;
- Elasticsearch отвечает за поиск;
- каждый слой включается и настраивается отдельно.

## 3. Redis В Этом Проекте Используется Не Как Источник Истины, А Как Быстрый Временный Слой

Это особенно хорошо видно в [`generation.py`](../../../apps/backend/app/services/generation.py#L27).

### Подключение к Redis

В конструкторе `GenerationService` Redis-клиент создаётся в строках `31-34`.

Это означает, что сервис генерации использует Redis как внешнее быстрое хранилище для своих runtime-задач.

### Блокировка На Очередь Генерации

В строках `211-244` используется `self.redis.lock(...)`.

Зачем это нужно:
- если несколько запросов одновременно захотят продвинуть очередь генерации;
- нам нужно, чтобы один поток не мешал другому;
- Redis-lock помогает координировать это между конкурентными операциями.

Это уже не "кэш страницы", а инфраструктурный use case Redis как coordination layer.

### Временный Runtime-Слепок Job

В строках `272-280` сервис кладёт в Redis состояние job:
- `status`;
- `progress`;
- `result_url`.

И задаёт `expire(..., 3600)`, то есть TTL на час.

Почему это не основная база:
- окончательное состояние job всё равно хранится в PostgreSQL;
- Redis здесь только ускоряет быстрый runtime-доступ к короткоживущему слепку.

### Cooldown Для Обновления Очереди

В строках `557-600` Redis используется ещё и для cooldown-механики:
- `get(...)` читает время последнего refresh;
- `set(..., ex=...)` ставит новое значение с автоматическим истечением;
- это удобно, потому что Redis хорошо подходит для коротких TTL-состояний.

## 4. Elasticsearch В Этом Проекте Используется Как Отдельный Поисковый Индекс

### Создание Async-Клиента

В [`elasticsearch.py`](../../../apps/backend/app/integrations/elasticsearch.py#L1):
- `get_elasticsearch_client()` в строках `9-16` лениво создаёт `AsyncElasticsearch`;
- `close_elasticsearch_client()` в строках `19-23` закрывает клиент при завершении приложения.

Важно, что индексирование можно выключить:
- в строках `11-13` клиент вообще не создаётся, если `ENABLE_SEARCH_INDEXING=false`.

### Создание Индексов При Старте Приложения

В [`main.py`](../../../apps/backend/app/main.py#L20) внутри `lifespan` в строке `26` вызывается:
- `await search_service.ensure_indices()`

А сама логика находится в [`search.py`](../../../apps/backend/app/services/search.py#L9):
- `ensure_indices()` в строках `10-46`;
- индекс `projects` создаётся в строках `15-26` и `41-42`;
- индекс `blog_posts` создаётся в строках `27-38` и `43-44`.

Это значит:
- поисковая структура создаётся отдельно от таблиц PostgreSQL;
- приложение заботится о наличии индексов при старте.

### Индексация Данных После Изменений

Когда администратор создаёт или обновляет проект, backend не только пишет в PostgreSQL, но и обновляет поисковый индекс.

Это видно в [`projects.py`](../../../apps/backend/app/api/routes/projects.py#L45):
- после `session.commit()` в строке `54` вызывается `search_service.index_project(project)` в строке `55`;
- при обновлении то же самое происходит в строках `73-74`;
- при удалении вызывается `search_service.delete_document("projects", slug)` в строке `90`.

То же самое есть и для блог-постов в [`blog_posts.py`](../../../apps/backend/app/api/routes/blog_posts.py#L47):
- создание и индексация в строках `55-57`;
- обновление и переиндексация в строках `74-76`;
- удаление документа из индекса в строке `92`.

### Поисковый Запрос

Собственно поиск реализован в [`search.py`](../../../apps/backend/app/services/search.py#L95):
- метод `search(...)` в строках `95-110`;
- используется `multi_match` в строках `102-107`;
- поиск идёт сразу по нескольким полям;
- полям `title_ru` и `title_en` дан больший вес через `^3`.

Это уже типичный пример полнотекстового поиска:
- не просто `WHERE title = ...`;
- а поиск по нескольким текстовым полям с относительной важностью.

## 5. Пакеты Для Redis И Elasticsearch Явно Установлены В Backend

А переменные окружения для них описаны в [`.env.example`](../../../.env.example#L12):

- `ELASTICSEARCH_URL` в строке `17`;
- `ENABLE_SEARCH_INDEXING` в строке `41`.

## Что Важно Понять Про Роли Слоёв В Этом Проекте

Если собрать всё вместе:

- `PostgreSQL` хранит основную бизнес-истину;
- `Redis` ускоряет и координирует временные runtime-сценарии;
- `Elasticsearch` помогает искать по контенту лучше, чем обычный SQL-фильтр.

Это хороший пример того, что зрелый backend почти никогда не сводится к одной единственной базе.

## Anti-Patterns Рядом С Этой Темой

### Anti-pattern 1. Пытаться хранить основную бизнес-истину в Redis

Redis удобен и быстрый, но его задача обычно не в том, чтобы быть главным и единственным хранилищем критичных бизнес-данных.

Если начать хранить в нём всё подряд, можно получить:
- сложное восстановление после сбоев;
- нечёткие границы ответственности;
- путаницу между временным и постоянным состоянием.

### Anti-pattern 2. Использовать Elasticsearch как замену всей базе данных

Elasticsearch отлично ищет, но он не заменяет полноценную транзакционную модель backend-системы.

Если пытаться делать из него главный storage, быстро начнутся вопросы:
- где настоящая система истины;
- как поддерживать консистентность;
- как удобнее делать транзакционные записи и связи.

### Anti-pattern 3. Искать всё через SQL `LIKE`

Для маленьких таблиц это ещё может работать.

Но как только контента и сценариев поиска становится больше, `LIKE '%слово%'` начинает проигрывать:
- по удобству;
- по релевантности;
- по качеству поиска;
- по масштабируемости.

### Anti-pattern 4. Класть в Redis всё подряд "на всякий случай"

Кэш должен иметь понятную причину существования.

Хороший вопрос перед добавлением Redis-ключа:
- что именно это ускоряет;
- сколько живёт это значение;
- можно ли восстановить его без потери истины;
- кто отвечает за очистку и срок жизни.

## Что Должно Остаться В Голове После Этой Главы

После этой главы важно удержать простую картину:

- база данных хранит истину;
- Redis ускоряет и координирует временные состояния;
- Elasticsearch обслуживает поиск по тексту;
- зрелая система может использовать несколько storage-слоёв одновременно, если у каждого есть понятная роль.

## Что Дальше

Следующий логичный шаг курса — глава про `OOP`, `SOLID` и `clean architecture`, потому что после данных, кэша и поиска уже хорошо видно, зачем вообще backend делят на слои и почему "просто писать всё в одном роуте" плохо растёт.
