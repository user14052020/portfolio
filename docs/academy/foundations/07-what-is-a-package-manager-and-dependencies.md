# Что Такое Package Manager И Зависимости

Это следующий фундаментальный шаг курса.

После этой главы слова `dependency`, `package manager`, `npm`, `pip`, `requirements.txt`, `package.json`, `lock file` и `semver` уже не должны казаться просто набором модных терминов.

Именно здесь становится понятно:
- откуда в проекте берутся внешние библиотеки;
- кто решает, какие версии библиотек поставить;
- почему один проект запускается у одного разработчика, но ломается у другого;
- зачем вообще нужны файлы зависимостей.

## Что Это Академически

### Зависимость

Зависимость — это внешний программный компонент, библиотека или пакет, от которого зависит работа другой программы.

Иначе говоря:
- ваш код использует не только то, что написали вы;
- он может опираться на код, написанный другими;
- этот внешний код и называется зависимостью.

### Пакет

Пакет — это единица поставки программного кода и сопутствующих метаданных.

Пакет обычно включает:
- исходный код;
- имя;
- версию;
- список собственных зависимостей;
- инструкции по установке;
- иногда скрипты, типы, бинарные файлы и конфигурацию.

### Package Manager

Package manager — это программный инструмент, который управляет пакетами и зависимостями.

Он обычно умеет:
- скачивать пакеты;
- разрешать дерево зависимостей;
- устанавливать нужные версии;
- обновлять пакеты;
- удалять пакеты;
- фиксировать зависимости проекта.

### Dependency Graph

Граф зависимостей — это структура, показывающая не только прямые зависимости проекта, но и зависимости этих зависимостей, и так далее по цепочке.

То есть если вы устанавливаете один пакет, то реально может приехать не один пакет, а целое дерево.

### Lock File

Lock file — это файл, в котором фиксируется точный, воспроизводимый набор установленных зависимостей и их версий.

Он нужен, чтобы:
- на разных машинах получался одинаковый dependency tree;
- сборка была повторяемой;
- проект не ломался внезапно из-за того, что “вчера подтянулась новая минорная версия”.

### SemVer

SemVer, или semantic versioning, — это соглашение о версиях пакетов в формате:

```text
MAJOR.MINOR.PATCH
```

Где:
- `MAJOR` — несовместимые изменения;
- `MINOR` — новые возможности без ожидаемого слома совместимости;
- `PATCH` — исправления багов без изменения API по смыслу.

## Что Это Простыми Словами

Если по-человечески:

- зависимость — это чужой готовый инструмент, который вы взяли себе в проект;
- package manager — это завхоз, который умеет эти инструменты находить, приносить и раскладывать;
- lock file — это опись того, что именно лежит на складе;
- semver — это способ понять, насколько опасно обновлять инструмент.

Пример из жизни:

если вы строите мастерскую, то:
- ваши собственные чертежи — это ваш код;
- купленный шуруповёрт — это зависимость;
- список того, какие инструменты закуплены и каких моделей, — это lock file;
- кладовщик, который это всё закупает и ведёт, — это package manager.

## Почему Это Реализовано Именно Так

Ни frontend, ни backend в реальном проекте почти никогда не пишутся “с нуля в полном вакууме”.

Почему:
- React уже решает большой класс задач UI;
- FastAPI уже решает большой класс задач HTTP API;
- SQLAlchemy уже решает доступ к БД;
- Redis-клиенты, HTTP-клиенты, валидаторы, анимации, иконки и типы уже существуют.

Поэтому инженерный мир давно пришёл к тому, что проект:
- хранит декларацию зависимостей;
- использует инструмент, который умеет их установить;
- по возможности фиксирует точные версии.

## Какие Anti-Patterns Рядом Существуют

### Anti-pattern 1. “Я просто поставлю последнюю версию всего”

Это кажется удобным, но в реальном проекте опасно.

Почему:
- пакет мог выпустить несовместимое изменение;
- пакет мог поменять API;
- новая версия могла конфликтовать с другим пакетом.

### Anti-pattern 2. Путать dependency с import

Не каждый `import` означает внешнюю зависимость.

Например:
- `import { env } from "@/shared/config/env";` — это импорт вашего локального модуля;
- `import { z } from "zod";` — это импорт внешней зависимости.

### Anti-pattern 3. Думать, что lock file не нужен

Без lock file разные разработчики могут получить:
- разные версии транзитивных зависимостей;
- разные баги;
- разное поведение проекта;
- разную сборку между локальной машиной и сервером.

### Anti-pattern 4. Думать, что `package.json` и `requirements.txt` — одно и то же

Они похожи по роли, но живут в разных экосистемах:
- `package.json` относится к Node.js / npm-миру;
- `requirements.txt` относится к Python / pip-миру.

### Anti-pattern 5. Не различать прямые и транзитивные зависимости

Прямые зависимости — это те, которые вы указали сами.

Транзитивные — это зависимости ваших зависимостей.

Именно поэтому даже маленький проект может реально установить намного больше пакетов, чем видно в одном файле декларации.

## npm, pip, package.json, requirements.txt И Lock Files

### npm

`npm` — это package manager в экосистеме Node.js / JavaScript / TypeScript.

Он работает с:
- `package.json`;
- `node_modules`;
- `package-lock.json`.

Типичные команды:
- `npm install`
- `npm run dev`
- `npm run build`
- `npm uninstall <package>`

### pip

`pip` — это пакетный менеджер Python-экосистемы.

Он работает с:
- Python-пакетами;
- индексами пакетов;
- файлами вроде `requirements.txt`.

Типичные команды:
- `pip install -r requirements.txt`
- `pip install fastapi`
- `pip freeze`

### package.json

`package.json` — это манифест frontend-проекта в Node.js-экосистеме.

В нём обычно лежат:
- имя пакета;
- версия;
- scripts;
- dependencies;
- devDependencies;
- иногда engines, exports, linting metadata и другое.

### requirements.txt

`requirements.txt` — это файл, в котором обычно перечислены Python-зависимости проекта, часто в формате:

```text
package==exact_version
```

Это очень простой и понятный способ зафиксировать набор зависимостей backend-сервиса.

### Lock Files

В JavaScript-мире lock file часто бывает:
- `package-lock.json`
- `yarn.lock`
- `pnpm-lock.yaml`

В Python-мире часто встречаются:
- `poetry.lock`
- `Pipfile.lock`
- `uv.lock`

Идея у всех одна:
- не просто сказать “мы примерно хотим вот эти зависимости”;
- а зафиксировать точный результат разрешения зависимостей.

## Что Реализовано В Этом Проекте

### 1. У фронта есть package.json

В проекте frontend использует:

- `apps/frontend/package.json`

Там видно:
- имя пакета `portfolio-frontend`;
- scripts `dev`, `build`, `start`, `lint`;
- runtime dependencies вроде `next`, `react`, `zod`, `framer-motion`;
- devDependencies вроде `typescript`, `eslint`, `tailwindcss`.

Это означает, что frontend проекта живёт в Node.js-экосистеме и использует npm-подход к зависимостям.

### 2. У бэка есть requirements.txt

В проекте backend использует:

- `apps/backend/requirements.txt`

Там видно точные Python-зависимости:
- `fastapi==0.115.12`
- `uvicorn[standard]==0.34.0`
- `sqlalchemy[asyncio]==2.0.39`
- `asyncpg==0.30.0`
- `alembic==1.15.2`
- `redis==5.2.1`
- `httpx==0.28.1`

Это означает, что backend проекта живёт в Python-экосистеме и использует pip-style декларацию зависимостей.

### 3. У фронта и бэка разные package managers, и это нормально

В одном проекте одновременно существуют:
- frontend-мир с `npm`;
- backend-мир с `pip`.

Это не “хаос”, а нормальная ситуация для full-stack системы, если роли слоёв понятны.

### 4. В проекте сейчас нет lock file

По текущему состоянию репозитория lock file не найден:
- нет `package-lock.json`;
- нет `yarn.lock`;
- нет `pnpm-lock.yaml`;
- нет `poetry.lock`;
- нет `Pipfile.lock`;
- нет `uv.lock`.

Это важное инженерное наблюдение.

Что это значит:
- зависимости декларированы;
- но точный воспроизводимый dependency snapshot сейчас не зафиксирован отдельным lock-файлом.

Для учебника это полезно, потому что мы видим реальный неидеальный production-нюанс, а не только “красивую теорию”.

### 5. У фронта уже есть смешение точных и диапазонных версий

В `apps/frontend/package.json` видно, что некоторые пакеты записаны как:

- `next`: `14.2.26`
- `react`: `18.3.1`

А другие — через `^`:

- `zod`: `^3.24.3`
- `framer-motion`: `^12.7.4`
- `typescript`: `^5.8.2`

Это хороший проектный пример для разговора про semver:
- точная версия фиксирует пакет жёстче;
- `^` разрешает обновления в рамках совместимого диапазона по semver-логике.

### 6. У backend зависимости в requirements.txt закреплены жёстко

В `apps/backend/requirements.txt` зависимости заданы через `==`.

Это означает:
- backend явно просит точные версии пакетов;
- риск неожиданного обновления ниже;
- воспроизводимость выше, чем при свободных диапазонах.

Но при этом:
- полноценного lock-файла уровня всего dependency graph всё равно нет;
- транзитивные зависимости могут не быть зафиксированы так строго, как при отдельном lock-подходе.

## Разбор На Примере

### Простой Вымышленный Пример

Представим маленький frontend-проект `weather-ui`.

Его `package.json` может выглядеть так:

```json
{
  "name": "weather-ui",
  "private": true,
  "dependencies": {
    "react": "18.3.1",
    "zod": "^3.24.3"
  }
}
```

Что это значит:
- проект сам напрямую просит `react` и `zod`;
- npm установит их;
- у этих пакетов тоже будут свои зависимости;
- итоговое дерево может оказаться заметно больше.

Если рядом есть `package-lock.json`, то результат установки фиксируется точнее.

### Пример Из Текущего Проекта

В нашем проекте:

- frontend зависит от `next`, `react`, `react-dom`, `zod`, `framer-motion`, `@mantine/*` и других пакетов;
- backend зависит от `fastapi`, `sqlalchemy`, `asyncpg`, `alembic`, `redis`, `httpx` и других библиотек.

То есть когда вы поднимаете весь проект, вы на самом деле поднимаете:
- не только свой код;
- но и большой слой внешнего кода, без которого проект не сможет собраться и запуститься.

Это очень важная мысль для новичка:

современный проект — это почти всегда ваш код плюс управляемый граф внешних зависимостей.

## Команды

### Установить frontend-зависимости

```bash
cd apps/frontend
npm install
```

### Установить backend-зависимости

```bash
cd apps/backend
pip install -r requirements.txt
```

### Посмотреть, какие зависимости declared у фронта

```bash
Get-Content apps/frontend/package.json
```

### Посмотреть backend-зависимости

```bash
Get-Content apps/backend/requirements.txt
```

### Найти lock-файлы в репозитории

```bash
rg --files | rg "(package-lock\\.json|yarn\\.lock|pnpm-lock\\.yaml|poetry\\.lock|Pipfile\\.lock|uv\\.lock)$"
```
