# Что Такое База Данных

Это следующий фундаментальный шаг курса.

После этой главы слова `база данных`, `таблица`, `строка`, `колонка`, `схема`, `запрос`, `индекс`, `транзакция` и `миграция` уже не должны звучать как абстрактный жаргон.

Именно здесь становится понятно:
- зачем проекту вообще нужно постоянное хранилище;
- почему данные нельзя просто держать в памяти Python-процесса;
- как backend узнает, где лежат чат-сообщения, пользователи, проекты и задачи генерации;
- почему изменение структуры таблиц тоже должно быть управляемым кодом.

## Что Это Академически

### База Данных

База данных — это организованная совокупность данных, предназначенная для долговременного хранения, поиска, изменения и совместного использования информационной системой.

Если сказать точнее:
- данные в базе хранятся не случайно, а по правилам;
- к ним можно выполнять формализованные операции чтения и записи;
- база данных должна сохранять целостность данных между разными запросами, процессами и перезапусками сервиса.

### Таблица

Таблица — это структурированное представление набора однотипных записей в реляционной базе данных.

Обычно таблица описывает один тип сущности:
- `users`;
- `projects`;
- `generation_jobs`;
- `chat_messages`.

### Строка

Строка, или row, — это одна конкретная запись в таблице.

Например, одна строка в `generation_jobs` может описывать одну задачу генерации изображения.

### Колонка

Колонка, или column, — это именованное поле в таблице, описывающее один атрибут записи.

Например:
- `id`;
- `status`;
- `prompt`;
- `created_at`.

### Схема

Схема базы данных — это структура данных системы: какие есть таблицы, какие у них колонки, типы, ограничения, связи и индексы.

Проще говоря, схема отвечает на вопрос:

```text
как именно организованы данные в хранилище
```

### Запрос

Запрос, или query, — это формальное обращение к базе данных с целью получить, создать, изменить или удалить данные.

Примеры намерений запроса:
- получить все опубликованные проекты;
- найти пользователя по email;
- сохранить новое сообщение в чате;
- обновить статус generation job.

### Индекс

Индекс — это вспомогательная структура данных, ускоряющая поиск и сортировку записей по определенным полям.

Цена индекса:
- чтение часто становится быстрее;
- запись может стать немного тяжелее;
- индекс занимает дополнительное место.

### Транзакция

Транзакция — это логически завершенная группа операций над базой данных, которая должна выполниться либо полностью, либо не выполниться совсем.

Это нужно, чтобы не оставлять систему в “полусломанном” состоянии.

### Миграция

Миграция — это управляемое изменение схемы базы данных, описанное кодом.

С помощью миграций команда может:
- создать новые таблицы;
- добавить колонку;
- добавить индекс;
- изменить ограничение;
- откатить изменение назад.

## Что Это Простыми Словами

Если по-человечески:

- база данных — это не просто “место, где что-то лежит”, а организованный склад;
- таблица — это один тип коробок на складе;
- строка — это одна конкретная коробка;
- колонка — это подпись на коробке: что в ней хранится;
- индекс — это быстрый указатель, чтобы не перебирать весь склад руками;
- транзакция — это правило “или делаем всю операцию целиком, или не делаем ничего”;
- миграция — это официальный ремонт склада по чертежу, а не хаотичное ломание стен.

Если упростить еще сильнее:

- память процесса — это рабочий стол;
- база данных — это архив и картотека;
- файл `.py` с моделью — это описание того, как выглядит карточка;
- миграция — это инструкция, как перестроить архив, если карточки изменились.

## Почему Это Реализовано Именно Так

Проект `portfolio` — это не один скрипт, который сделал что-то и завершился.

У нас есть:
- frontend;
- backend;
- PostgreSQL;
- Redis;
- Elasticsearch;
- отдельные runtime-компоненты вроде `ComfyUI` и `vLLM`.

Если бы данные жили только в памяти backend-процесса:
- они бы исчезали при рестарте контейнера;
- история чата бы терялась;
- generation jobs бы пропадали;
- админка не могла бы показывать стабильное состояние системы;
- несколько процессов не имели бы общего источника истины.

Поэтому в проекте есть отдельная база данных, а backend работает с ней через слой ORM и сессий.

## Какие Anti-Patterns Рядом Существуют

### Anti-pattern 1. Держать важные данные только в памяти

Это подходит для временных вычислений, но не для бизнес-данных.

Если данные лежат только в переменной Python-процесса:
- они пропадут после рестарта;
- другой процесс их не увидит;
- история системы станет неустойчивой.

### Anti-pattern 2. Думать, что база данных = “одна таблица”

В реальной системе база данных почти всегда состоит из набора связанных таблиц.

Например, в нашем проекте есть не только `generation_jobs`, но и:
- `users`;
- `projects`;
- `blog_posts`;
- `uploaded_assets`;
- `chat_messages`.

### Anti-pattern 3. Игнорировать индексы

Новичку часто кажется, что база “сама как-нибудь найдет строку”.

Но без индексов:
- поиск по большим таблицам становится медленным;
- сортировка и фильтрация могут упираться в полный перебор;
- система хуже масштабируется.

### Anti-pattern 4. Менять схему базы вручную на проде

Если инженер “просто зайдет в базу и добавит колонку руками”, то:
- изменение не будет описано в коде;
- другая машина не узнает о нем автоматически;
- staging, dev и prod могут разъехаться по структуре.

Именно поэтому нужны миграции.

### Anti-pattern 5. Делать несколько связанных записей без транзакции

Если операция должна записать несколько связанных сущностей, а посередине произошла ошибка, система может остаться в неконсистентном состоянии.

Пример плохого сценария:
- chat message уже сохранен;
- generation job еще не сохранена;
- UI думает, что все прошло успешно;
- в базе полусломанная картина.

## Что Реализовано В Этом Проекте

### 1. В проекте есть отдельный контейнер PostgreSQL

Это видно в [docker-compose.yml](../../../docker-compose.yml): `2-18`.

Там описано:
- сервис `postgres`;
- образ `postgres:16-alpine`;
- переменные `POSTGRES_DB`, `POSTGRES_USER`, `POSTGRES_PASSWORD`;
- volume для постоянного хранения данных;
- healthcheck через `pg_isready`.

Это важная инженерная мысль:
- база данных живет как отдельный runtime-компонент;
- backend не “заменяет” базу;
- backend подключается к ней как клиент.

### 2. Backend подключается к БД через async engine и async session

Это видно в [session.py](../../../apps/backend/app/db/session.py): `10-16`.

Там:
- создается engine через `create_async_engine(...)`;
- создается фабрика сессий `async_sessionmaker(...)`;
- dependency `get_db_session()` выдает `AsyncSession`.

Это значит:
- backend не пишет SQL “прямо в каждом роуте” как попало;
- у него есть централизованный способ получить соединение с базой;
- доступ к базе уже встроен в архитектуру приложения.

### 3. Таблица generation_jobs описана как ORM-модель

Это видно в [generation_job.py](../../../apps/backend/app/models/generation_job.py): `10-45`.

Здесь очень наглядно видны основные термины:
- таблица: `__tablename__ = "generation_jobs"`;
- индексы: строки `12-16`;
- колонки: строки `18-42`;
- внешний ключ: `ForeignKey("uploaded_assets.id")` в строке `31`;
- JSON-колонки: строки `38-39`;
- связи ORM: строки `44-45`.

На этом одном файле уже можно понять:
- как объект в Python отображается в таблицу БД;
- что одна таблица хранит и простые поля, и enum-поля, и JSON, и связи;
- почему структура базы — это часть кода, а не “магия где-то в Postgres”.

### 4. Начальная схема базы создается миграцией

Это видно в [202603300001_initial.py](../../../apps/backend/alembic/versions/202603300001_initial.py): `32-256`.

Особенно полезно посмотреть:
- создание enum-типов: `32-40`;
- создание таблиц: начиная с `42`;
- создание `generation_jobs`: `226-256`.

Это и есть миграция в реальной жизни:
- не “кто-то где-то руками создал таблицу”;
- а зафиксированная инструкция в репозитории;
- которую можно воспроизвести на другой машине.

### 5. В проекте база хранит не только тексты, но и связи между сущностями

На примере `generation_jobs` видно, что база данных хранит:
- идентификатор job;
- статус;
- промпт;
- рекомендации;
- связь с загруженным ассетом;
- внешний `external_job_id`;
- прогресс;
- служебные поля времени.

То есть база нужна не просто как “блокнот со строками”, а как структурированный слой состояния всей системы.

## Таблица, Строка, Колонка И Индекс На Простом Примере

Представим вымышленную таблицу `books`.

Таблица:

```text
books
```

Колонки:
- `id`
- `title`
- `author`
- `published_year`

Одна строка:

```text
id=1, title="Tailoring Basics", author="Mira Vale", published_year=2024
```

Если книг мало, можно искать без индекса.

Если книг очень много, полезно добавить индекс, например по `author`, чтобы запрос вида:

```text
найти все книги Mira Vale
```

выполнялся быстрее.

## Разбор На Примере Текущего Проекта

Когда пользователь пишет стилисту сообщение, упрощенная цепочка выглядит так:

1. frontend отправляет HTTP-запрос на backend;
2. backend валидирует данные;
3. backend может сохранить chat message;
4. backend может создать generation job;
5. позже backend читает эти записи обратно и показывает их в UI.

Значит база данных здесь нужна для того, чтобы:
- состояние чата не терялось;
- generation jobs жили дольше одного запроса;
- админка видела накопленное состояние;
- система могла пережить перезапуск контейнера.

Без базы данных весь этот слой был бы слишком хрупким.

## Чем База Данных Отличается От Соседних Понятий

### База данных vs файл

Файл тоже может хранить данные, но:
- база данных умеет запросы;
- база данных умеет индексы;
- база данных умеет транзакции;
- база данных лучше подходит для многопользовательской и многопроцессной работы.

### База данных vs память процесса

Память процесса быстрая, но временная.

База данных:
- медленнее памяти;
- но долговечнее;
- разделяется между процессами;
- сохраняет данные после рестарта.

### База данных vs кэш

Кэш нужен, чтобы ускорять доступ к данным.

База данных нужна, чтобы хранить основной, устойчивый источник истины.

В нашем проекте:
- PostgreSQL — основной источник истины;
- Redis — отдельный компонент для задач кэша и быстрого вспомогательного доступа.

Подробно это будет в отдельной главе про кэш и поиск.

## Команды

### Посмотреть, что сервис PostgreSQL описан в docker-compose

```bash
Get-Content docker-compose.yml
```

### Посмотреть код подключения backend к базе

```bash
Get-Content apps/backend/app/db/session.py
```

### Найти, где в проекте используются модели SQLAlchemy

```bash
rg -n "__tablename__|mapped_column|ForeignKey|relationship" apps/backend/app/models
```

### Найти миграции Alembic

```bash
rg --files apps/backend/alembic/versions
```

### Если проект уже поднят, посмотреть таблицы внутри PostgreSQL

```bash
docker compose exec postgres psql -U "$env:POSTGRES_USER" -d "$env:POSTGRES_DB" -c "\\dt"
```
