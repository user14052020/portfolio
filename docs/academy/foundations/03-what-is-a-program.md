# Что Такое Компьютерная Программа

Это первая фундаментальная глава курса.

Она нужна, чтобы дальше слова вроде `frontend`, `backend`, `Docker`, `API`, `процесс`, `сервер` и `приложение` не висели в воздухе.

## Что Это Академически

Компьютерная программа — это формализованная последовательность инструкций и данных, предназначенная для выполнения вычислительной системой с целью обработки входных данных и получения результата.

Если говорить строго, у программы почти всегда есть:
- инструкции;
- данные;
- вход;
- выход;
- состояние;
- среда выполнения.

## Что Это Простыми Словами

Программа — это набор точных указаний для компьютера:

- что принять на вход;
- что с этим сделать;
- что вернуть на выход;
- что помнить между шагами.

Примеры из жизни:
- калькулятор считает числа;
- браузер открывает страницы;
- чат-ассистент принимает сообщение и показывает ответ;
- генератор изображений получает prompt и строит картинку.

## Почему Это Реализовано Именно Так

Весь текущий проект на самом деле состоит не из "одного сайта", а из нескольких программ, которые работают вместе:

- frontend-приложение на Next.js;
- backend-приложение на Python;
- PostgreSQL как отдельная программа-сервер базы данных;
- Redis как отдельная программа-сервис;
- Elasticsearch как отдельная программа-поисковый движок;
- ComfyUI как отдельная программа для генерации изображений;
- vLLM как отдельная программа для text inference.

Поэтому ещё до разговора про архитектуру важно понять базовую мысль:

`проект = это не папка с файлами, а набор программ, процессов, данных и связей между ними`.

## Какие Anti-Patterns Рядом Существуют

### Anti-pattern 1. Думать, что программа — это только код

Это неверно.

Программа — это:
- код;
- данные;
- конфигурация;
- runtime;
- вход и выход;
- иногда сеть, база данных и файловая система.

### Anti-pattern 2. Не различать программу, процесс и сервис

Это разные вещи:
- программа — это описание того, что можно запустить;
- процесс — это уже запущенный экземпляр программы;
- сервис — это программа или набор процессов, которые выполняют полезную системную или прикладную роль.

### Anti-pattern 3. Смотреть на проект как на "набор экранов"

Такой взгляд мешает понимать систему целиком.

Даже самый красивый интерфейс бессмысленен без:
- backend-логики;
- хранения данных;
- сетевого обмена;
- фоновых задач;
- окружения, где всё это реально живёт.

## Что Реализовано В Этом Проекте

Прямо в текущем проекте мы уже видим несколько типов программ.

### 1. Frontend-программа

Она живёт в:
- `apps/frontend`

Её задача:
- показать страницу;
- принять ввод пользователя;
- отправить запрос на backend;
- отобразить ответ;
- отследить прогресс генерации.

### 2. Backend-программа

Она живёт в:
- `apps/backend`

Её задача:
- принять HTTP-запрос;
- проверить входные данные;
- запустить бизнес-логику;
- сохранить данные;
- вернуть ответ;
- координировать генерацию и интеграции.

### 3. Программы инфраструктуры

Они подключаются через `docker-compose.yml`:
- Postgres;
- Redis;
- Elasticsearch.

Это тоже полноценные программы, просто не написанные нами с нуля.

### 4. Внешние AI runtime-компоненты

Отдельно в этом проекте работают:
- `vLLM`
- `ComfyUI`

Они тоже программы:
- получают вход;
- делают вычисления;
- выдают результат;
- ведут логи;
- занимают память и GPU.

## Что Ещё Можно Улучшить

Даже на уровне понимания программы здесь можно закладывать инженерное мышление сильнее.

Например, дальше в курсе важно отдельно показать:
- где заканчивается одна программа и начинается другая;
- как программы общаются по сети;
- как один и тот же код превращается в разные процессы;
- как окружение влияет на поведение программы;
- почему `.env`, Docker и system services так важны для запуска.

## Разбор На Примере

### Простой вымышленный пример

Возьмём воображаемую программу "калькулятор суммы двух чисел".

Её можно описать так:

- вход: два числа;
- инструкция: сложить их;
- состояние: временно хранить значения в памяти;
- выход: одно итоговое число.

Псевдокод:

```text
прочитать a
прочитать b
result = a + b
показать result
```

Это уже программа, даже если она очень маленькая.

### Пример из текущего проекта

Когда пользователь пишет сообщение в чат:

1. frontend-программа принимает текст;
2. frontend отправляет HTTP-запрос;
3. backend-программа получает запрос;
4. backend решает, нужен ли ответ от `vLLM`, запуск `ComfyUI` или работа с БД;
5. backend возвращает JSON;
6. frontend показывает новый ответ;
7. при генерации картинки отдельная программа `ComfyUI` строит изображение.

То есть один пользовательский сценарий на сайте — это уже работа сразу нескольких программ.

## Команды

### Посмотреть корневые точки входа проекта

```bash
rg --files | rg "package.json|main.py|docker-compose.yml|\\.env.example"
```

### Посмотреть, из каких крупных частей состоит репозиторий

```bash
rg --files apps docs | Select-Object -First 50
```

### Найти Python entrypoints и frontend entrypoints

```bash
rg -n "FastAPI\\(|export default|export function|createTheme|docker compose" apps docs
```

## Мини-Упражнения

1. Попробуйте своими словами объяснить разницу между программой и файлом.
2. Назовите три отдельные программы, которые участвуют в работе этого проекта.
3. Опишите один пользовательский сценарий на сайте как цепочку из нескольких программ.
4. Подумайте, что произойдёт, если frontend-программа запустится, а backend — нет.
